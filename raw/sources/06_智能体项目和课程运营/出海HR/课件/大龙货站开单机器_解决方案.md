# 大龙货站"开单机器"解决方案

## 背景

成长型企业老板使用豆包个人版做工作自动化，遇到以下核心问题：
- AI 过于"善于表现"，约束词多了也摁不住
- 换个话题 AI 就跑偏
- 目标：做一个严格按 9 步引导用户填写信息的"开单机器"

## 一、豆包各版本能力差异与适用场景分析

### 1.1 三个版本的定位差异

| 维度 | 豆包个人版（App/网页） | 豆包企业版（火山引擎） | 豆包 API（直接调用） |
|------|----------------------|----------------------|---------------------|

| **话题控制** | 弱，容易被用户带跑 | 可配置话题边界 | 完全由调用方控制 |
| **状态记忆** | 单次对话有效 | 支持会话状态管理 | 自己维护状态 |
| **输出格式** | 随意，散文/列表混搭 | 可配置 JSON 输出 | 完全自定义 |
| **费用** | 免费（有次数限制） | 按 token 计费 | 按 token 计费 |
| **适用场景** | 日常问答、灵感激发 | 企业客服、知识库问答 | 流程化任务、工作流嵌入 |

### 1.2 哪个版本适合"开单机器"场景？

**结论：API 方案最合适，个人版不推荐，企业版可作为过渡。**

原因：
- "开单机器"本质是一个**有限状态机**：用户只能在当前步骤回答当前问题，不允许跳步、不允许发散
- 个人版的对齐策略（RLHF）会让 AI 优先"让用户满意"而非"严格执行流程"，这与业务刚性需求矛盾
- 企业版虽然有话题控制能力，但 9 步引导的精度仍不如 API + 状态维护
- API 方案可以自己维护当前步骤状态，每次只允许用户输入当前步骤的内容

## 二、推荐方案

### 方案 A（推荐）：豆包 API + 状态机控制层

**架构：**

```
用户输入 → 状态机（判断当前步骤） → Prompt 构造 → API 调用 → 结果展示 → 状态更新
```

**核心思路：** 不要让 AI 自己判断"现在该问什么"，而是由程序控制流程，AI 只负责当前步骤的提问和字段校验。

**优点：**
- 流程绝对可控，不会跳步
- AI 只负责当前步骤的内容生成和简单校验
- 可以随时插入人工审核节点
- 支持复杂条件分支（如某些步骤根据上一步答案跳过后续步骤）

**缺点：**
- 需要一点开发工作（但门槛很低，用 Coze 工作流或腾讯云混元都可以零代码搭建）

---

### 方案 B（过渡）：Coze（扣子）工作流

如果团队没有开发资源，可以用 Coze（字节旗下，零代码 Bot 构建平台）：

- 使用**工作流**节点定义 9 个步骤的顺序
- 每个步骤用**LLM 节点**生成当前问题
- 用**表单节点**收集用户输入
- 工作流强制线性执行，无法跳步

**限制：** Coze 的工作流在用户输入层面仍是开放对话，AI 可能被用户引导到工作流外的对话中去。需要额外配置"话题边界"技能插件。

---

### 方案 C（不推荐但可行）：豆包企业版 + 知识库

- 配置知识库限定业务范围
- 开启"强制话题回归"策略
- **问题：** 企业版的对话管理仍是基于语义相似度判断，无法做到严格的步骤锁定

## 三、Prompt 优化建议：如何写一个能"摁住"AI的 Prompt

### 3.1 核心原则：把 AI 变成"填空题机器"而不是"对话机器人"

**BAD Prompt（太开放）：**
> "你现在是一个开单助手，请询问用户发货信息，包括收件人、地址、货物信息等。"

**问题：** AI 会自由发挥，可能问完一个字段就开始闲聊，或者问一堆"还需要其他信息吗"之类的话。

---

**GOOD Prompt（状态注入 + 严格边界）：**

```
## 角色
你是一个"开单助手"。你没有名字，没有情绪，不会闲聊，不回答与开单无关的问题。

## 当前状态
- 当前步骤：{step_number} / 9
- 当前步骤名称：{step_name}
- 当前步骤需要收集的字段：{field_name}
- 已收集的信息：{collected_data}

## 本步任务
只问用户 `{step_question}` 这一个问题。
不许问其他问题。
不许说"请问您还需要什么帮助吗"。
不许总结或复述已收集的信息。

## 回答规则
1. 只输出当前步骤的问题，不输出其他内容
2. 问题必须是简洁的疑问句
3. 如果用户输入与当前步骤无关，回答："请先回答当前问题：{step_question}"
4. 不管用户说什么，都不要离开当前步骤，除非用户正确回答了当前问题

## 输出格式
直接输出问题，不要前缀，不要解释，不要格式。
```

---

### 3.2 步骤级 Prompt 示例（以第 3 步"货物信息"为例）

```
## 当前步骤
步骤 3/9：货物信息

## 收集字段
- 货物名称
- 件数
- 重量（kg）
- 是否有电池/液体/易碎（是/否）

## 你的任务
用户已经完成了步骤 2（发货人信息）。
现在你需要向用户收集上述 4 个字段。
每次只问一个字段，等用户回答后再问下一个。
问问题时使用以下格式：
"请告诉我货物名称："

## 校验规则
- 货物名称：不能为空，不能超过 50 个字符
- 件数：必须是正整数
- 重量：必须是数字，最多 1 位小数
- 特殊货物：用户回答"是"或"否"

## 边界控制
- 用户说"跳过"→ 回答"货物信息为必填项，无法跳过"
- 用户转移话题 → 回答"请先完成货物信息填写：货物名称是？"
- 用户说"我不知道"→ 回答"货物名称请填写实际发货的商品名，例如'手机配件'"

## 不允许的行为
- 不允许说"好的，已经收到"
- 不允许说"接下来我们来填..."
- 不允许问与货物无关的问题
- 不允许输出表格或列表
```

---

### 3.3 让 AI"摁住"的 Prompt 技巧总结

| 技巧 | 说明 | 示例 |
|------|------|------|
| **注入当前状态** | 明确告知 AI 现在是第几步，减少它"自作主张" | `当前步骤：3/9` |
| **负面清单（Negative Prompt）** | 列出 AI 不许做的事，往往比正面清单更有效 | `不许问其他问题` |
| **默认回复（Default Response）** | 当用户跑偏时，AI 必须说什么 | `请先回答当前问题：xxx` |
| **输出格式强制约束** | 限制输出格式，减少 AI 自由发挥空间 | `只输出问题，不要前缀` |
| **禁止省略语** | 防止 AI 用"好的""了解"来结束对话开启闲聊 | `不允许说"好的"` |
| **条件分支预判** | 如果需要跳过某些步骤，提前在 prompt 里说明触发条件 | `如果货物重量<1kg，跳过步骤5` |

## 四、是否需要工作流 / RPA 等辅助手段？

### 4.1 什么时候需要工作流？

| 场景 | 推荐工具 |
|------|---------|
| 需要在 9 步中插入条件分支 | Coze 工作流 / Zapier |
| 开单完成后需要自动发邮件/通知 | 工作流（webhook 触发） |
| 需要对接 ERP / 物流系统 | API + 工作流 |
| 团队多人使用，需要权限管理 | 企业版 / Coze 团队空间 |
| 仅需要严格的步骤锁定 | **不需要**，纯 API + 状态机即可 |

### 4.2 什么时候不需要工作流？

如果只是：用户输入 → AI 校验 → 保存到数据库（Excel / Notion / 飞书表格），则**纯 API 方案即可**，不需要额外引入工作流。

### 4.3 RPA 的适用场景

RPA（如影刀、UiBot）适用于：
- 需要模拟鼠标键盘操作桌面软件
- 旧系统没有 API 接口，只能通过界面操作

对于"开单机器"，**不建议用 RPA**，因为：
- 开单信息最终需要存到某个数据库，API 直写数据库比 RPA 模拟人工输入更可靠
- RPA 稳定性差，界面改版就会崩溃

## 五、实施路径建议

### Phase 1（1-2天）：验证最小可行性
1. 在 Coze 上创建 Bot，用工作流定义 9 步顺序
2. 编写每步的 Prompt，用"步骤状态注入"方式控制 AI
3. 将用户输入保存到飞书多维表格（无需写代码）
4. 测试：找员工走一遍 9 步，看哪里会跑偏

### Phase 2（3-5天）：提升鲁棒性
1. 收集 Phase 1 测试中 AI 跑偏的 case
2. 针对性补充 Negative Prompt（禁止行为清单）
3. 增加输入校验逻辑（正则 / 范围校验）
4. 考虑迁移到 API 方案（如果 Coze 工作流频繁被用户绕开）

### Phase 3（长期）：自动化集成
1. 开单完成后自动推送Webhook到物流系统
2. 数据自动同步到内部ERP
3. 增加"历史订单查询"功能（用向量数据库做知识库）
