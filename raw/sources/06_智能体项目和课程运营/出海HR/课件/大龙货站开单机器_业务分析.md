# 大龙货站开单机器 — 业务需求分析

> 分析日期：2026-03-24  
> 分析对象：物流货站运单填写引导场景

---

## 一、背景还原

某成长型企业老板在探索"AI + 业务流程"自动化时，尝试用豆包个人版构建一个"开单机器"：让 AI 按照固定步骤，引导用户（很可能是货站员工或发货客户）一步步填写物流运单上的 9 个字段。

用户的核心痛点是：**AI 太"话多"、太发散，给了多个约束词之后依然"摁不住"，稍微用别的话题一引就跑偏了**。

---

## 二、业务场景评估

### 2.1 场景是否合理？

**结论：场景本身是合理的，但需求被错配到了通用对话 AI 上。**

理由如下：

| 维度 | 评估 |
|------|------|
| 业务需求真实性 | ✅ 真实。物流货站每天处理大量运单，引导式填写能降低错误率、提升效率 |
| 流程标准化程度 | ✅ 高。运单字段数量固定（9个），填写顺序可以固定，属于典型"表单引导"流程 |
| 目标用户痛点 | ✅ 明确。员工/客户不知道填什么、填错顺序、漏填字段 |
| AI 介入价值 | ✅ 成立。标准化引导可替代部分人工前台工作 |

### 2.2 "9个字段顺序填写"——业务逻辑是否清晰？

**结论：逻辑是清晰的，但"顺序"本身需要业务确认。**

物流运单的典型字段可能包括（示例）：

1. 发货人姓名 / 电话
2. 发货地址
3. 收货人姓名 / 电话
4. 收货地址
5. 货物名称
6. 货物重量 / 体积
7. 运费金额
8. 付款方式（现付 / 到付）
9. 备注

"严格按顺序"填写在技术上可以做到，但需要确认：
- 业务上是否存在"先填哪个都OK"的并行字段（如姓名和电话）？
- 是否存在"前置依赖"字段（如先选目的地才能显示对应运费规则）？
- 9个字段是**固定不变**的，还是存在不同类型运单（快递 / 货运 / 整车）导致字段差异？

## 三、这个场景是否适合用 AI 对话来做？

### 3.1 适合度评估

| 评估项 | 得分 | 说明 |
|--------|------|------|
| 流程可标准化程度 | ★★★★★ | 字段固定，步骤固定 |
| 用户输入自由度 | ★★☆☆☆ | 用户只需回答具体内容，不需要自由发挥 |
| 对话发散风险 | ★★☆☆☆ | 不需要发散，越收敛越好——这点与通用对话AI相反 |
| 错误容忍度 | ★★☆☆☆ | 字段填错、顺序乱了会影响后续物流环节 |
| 需要多轮对话判断 | ★★☆☆☆ | 基本是"问 → 答 → 记录 → 问下一个"，不需要复杂推理 |

**综合判断：这是一个"低对话需求、高流程约束"的场景。**

### 3.2 为什么通用对话 AI（如豆包）容易"摁不住"？

1. **目标函数冲突**：通用对话模型被训练为"尽可能给出有用、有趣、丰富的回答"，而开单机器需要"尽可能少说话、只问下一个问题"
2. **指令遵循的不确定性**：当用户说"等等，我再想想"或转移话题时，通用模型会"接住话题"并延展，而开单场景需要"礼貌地把话题拽回来"
3. **长 prompt 衰减**：给模型多个约束词时，各约束之间可能相互干扰，且模型对"约束边界"的理解有概率偏差
4. **缺乏状态追踪机制**：9个字段的填写状态、哪些已完成、哪些缺失，通用对话没有内置的"表单进度追踪"能力

## 四、用户期望与 AI 能力之间的 Gap

| 用户期望 | AI 现状 | Gap 描述 |
|----------|---------|----------|
| 严格按照 9 个字段顺序问 | 模型可能跳过、合并或调换顺序 | **流程控制能力不足** |
| 只回答运单相关信息 | 模型容易被其他话题带跑 | **话题锁定能力不足** |
| 给出约束词后立即生效 | 多个约束词叠加后效果衰减 | **复合指令遵循不稳定** |
| 忘记上一个问题时不重复 | 模型倾向于"接住"而非"追问" | **主动纠正/追回机制缺失** |
| 填完即可生成完整运单 | 模型只能输出文字，无法直接对接业务系统 | **与后端系统的打通能力为零** |

**核心 Gap：本场景需要的是一个"受约束的状态机"，而不是一个"自由的对话代理"。**

## 六、方案建议

### 6.1 不推荐：纯通用对话模型（如豆包、ChatGPT）

**原因**：架构层面的目标函数天然对立——你要的是一个"收敛的、守规矩的"工具，而通用模型是一个"发散的、有创造力的"工具。用 prompt engineering 可以缓解，但无法根除。

### 6.2 推荐方案

#### 方案 A（轻量级）：结构化 Prompt + 对话状态管理（ReAct / StateGraph）

- 用 LLM 负责**理解用户输入**和**生成下一问**
- 用外部代码管理**当前步骤、已填字段、缺失字段**
- 关键：LLM 只负责"NLU + NLGeneration"，流程控制由代码决定
- 适合：有简单开发能力，预算有限

#### 方案 B（工程化）：专用 AI 表单机器人（CoPAD / 扣子 / 轻自动化平台）

- 使用支持"节点流"的 AI 平台，拖拽式构建"问 → 记录 → 判断 → 下一问"逻辑
- 每个字段是一个独立节点，连接关系是固定的
- 适合：无开发团队，希望快速上线

#### 方案 C（最稳定）：规则引擎 + LLM 辅助（推荐用于生产）

- 核心逻辑：纯 if-then 规则引擎处理流程控制
- LLM 只用于：理解用户模糊表述（如"发到北京去"→ 解析为"收货地址：北京市"）
- 优势：100% 可控，无"跑偏"风险，响应速度快
- 适合：对稳定性和准确性有较高要求的生产环境

### 6.3 一句话总结

> **这不是一个"让 AI 更听话"的问题，而是一个"应该用结构化工具而不是对话代理来处理结构化流程"的问题。**

---

## 七、后续建议

1. **先确认 9 个字段的具体定义和业务规则**（是否有依赖关系？是否有必填/选填区分？）
2. **评估是否有现成的物流 SaaS 系统**，很多货站管理软件已有标准开单界面，AI 介入的必要性需要评估
3. **若决定自建，优先选方案 C**（规则引擎 + LLM 辅助），兼顾可控性和灵活性
4. **如果仅用于内部员工（而非客户）**，可以考虑做成"助手在旁"模式——员工开单，AI 实时纠错和补全，而非全程 AI 主导
