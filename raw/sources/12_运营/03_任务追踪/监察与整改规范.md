## 一、监察执行流程

### 触发时机（核心调整）

> **触发时机：每次分配任务到subagent时立即登记，而非任务完成时**

| 任务类型 | 登记时机 | 说明 |
|---------|---------|------|
| subagent任务 | ✅ 分配时立即 | sessions_spawn前必须先登记 |
| 手动任务 | ✅ 发布时 | 任务创建时登记 |
| 定时任务 | ✅ 触发时 | 自动创建记录 |

### 完整监察流程

```
[Step 0] 任务分配（触发时机）
    ↓
登记到任务协作看板（状态：进行中）
    ↓
sessions_spawn 启动 subagent
    ↓
[Step 1] subagent执行中
    ↓
[Step 2] subagent完成 → 更新看板状态
    ↓
[Step 3] 分发到对应群聊
    ↓
[Step 4] 任务关闭
```

### 重要调整：取消审核节点
> **任务完成后直接关闭，不再需要David审核**
> - 只需记录任务进展状态
> - 执行结果分发至对应群聊即可
> - 看板状态流转：进行中 → 已完成 → 任务关闭

### 看板字段完整填写

| 字段 | 必须填写 | 说明 |
|------|---------|------|
| 任务名称 | ✅ | 清晰描述 |
| 状态 | ✅ | 待处理/进行中/已完成 |
| 执行人 | ✅ | David Yang |
| 项目分类 | ✅ | 知识库建设/用友课程等 |
| 负责Agent | ✅ | 对应的subagent |
| 结果摘要 | ✅ | 完成后填写核心结论 |
| 审核状态 | ✅ | 待审核→已通过/需修改 |

| 触发条件 | 执行动作 |

|--------|------|------|---------|

## 三、知识库健康度检查标准

### 检查清单

| 检查项 | 标准 | 不符合处理 |
|--------|------|---------|
| 目录结构 | 与规划一致，无新增混乱目录 | 重新整理 |
| 文档命名 | 符合[类型]_[主题]_[版本]_[日期]格式 | 提醒修正 |
| 内容完整性 | 重要模块有实质内容，不是空壳 | 补充或删除 |
| 交叉引用 | 关联文档有引用关系记录 | 添加关联 |
| 更新时效 | 重大政策/案例24h内有更新 | 触发更新 |

```
🔴 严重问题：[列表]

```

## 五、判定标准

### 审核判定

| 你说 | 判定结果 | 后续动作 |
|------|---------|---------|
| OK / 通过 / 可以 | ✅ 已通过 | 关闭任务 |
| 需修改 / 不通过 | ⚠️ 需修改 | 重新执行 |
| 进行中 / 开始了 | 🔄 进行中 | 继续执行 |

### 紧急程度判定

| 情况 | 紧急程度 | 通知方式 |
|------|---------|---------|
| 审核超过3天未处理 | 🔴 高 | 立即通知你 |
| 任务执行失败 | 🔴 高 | 立即通知你 |
| 知识库严重问题 | 🟡 中 | 定期报告 |
| SOP需要更新 | 🟢 低 | 周常报告 |
