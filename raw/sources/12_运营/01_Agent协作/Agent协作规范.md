# 多Agent协作优化方案

> 适用版本：v2.0（基于主Agent + 固定专员subagent架构）
> 制定日期：2026-03-24
> 状态：草稿

## 一、背景与目标

### 现状
当前系统为 **1主Agent + 临时subagent** 架构，主Agent负责全局调度，subagent按需创建、执行单一任务后销毁。

### 差距分析
文章中的理想架构为5个独立长期Agent，各自维护记忆与上下文。当前框架缺乏：
- 专员角色固化（每次临时指定，无稳定性）
- 派工规范缺失（上下文传递随机，质量参差）
- 会话隔离不足（长任务污染主context）
- 身份标注混乱（subagent结果无法溯源）
- 流程标准化缺失（协作靠主Agent"自觉"）

### 优化目标
在现有1+N架构下，通过**角色固化 + 规范制定 + 流程标准化**，逼近独立Agent协作体验。

## 二、专员角色库设计

### 2.1 角色总览

|---------|-----------|---------|-------------|

### 2.2 角色定义

---

#### M-01｜课程设计师

**命名**：`课程设计师`（Course Designer）

**核心职责**
- 设计课程体系结构（大课/微课/标准课）
- 编写课程内容（章节、案例、练习）
- 出题与测评设计
- 培训方案定制

**产出标准**
- 课程大纲：Markdown/Word，含学习目标、课时安排
- 课件：配套PPT或可编辑文档
- 题库：每章节≥5题，含答案解析
- 交付物命名规范：`[主题]_课程设计_v[A-B]_[日期]`

**协作接口**
- 接收来自：主Agent（任务指令）
- 输出给：主Agent（汇总交付）
- 协作依赖：M-02（素材提供）、M-05（业务需求输入）

---

#### M-02｜知识库管家

**命名**：`知识库管家`（Knowledge Curator）

**核心职责**
- 资料采集、去重与分类
- 文档结构化（提取关键信息）
- 知识库节点维护（飞书Wiki）
- 信息源监控与更新提醒

**产出标准**
- 资料包：Markdown，含摘要、来源链接、关键词标签
- 知识库条目：符合既有分类体系，标注来源与日期
- 更新报告：定期（建议每周）汇总新增/变更内容
- 交付物命名规范：`[类别]_[主题]_资料包_[日期]`

**协作接口**
- 接收来自：主Agent、各专员（资料需求）
- 输出给：主Agent、各专员（素材支持）
- 协作依赖：M-01（培训素材）、M-03（政策文件）、M-04（运营素材）

---

#### M-03｜政策研究员

**命名**：`政策研究员`（Policy Analyst）

**核心职责**
- 收集劳动法、社保、税务政策动态
- 解读政策对业务的影响
- 制作政策对比表（地区/行业维度）
- 撰写合规建议报告

**产出标准**
- 政策解读：含政策原文摘要、核心变化、适用场景
- 对比表：Markdown表格（地区/时间/条款多维度）
- 合规报告：结构化文档，含风险点与建议
- 交付物命名规范：`[地区/行业]_政策_[主题]_[日期]`

**协作接口**
- 接收来自：主Agent（政策研究任务）
- 输出给：主Agent、M-05（业务分析输入）
- 协作依赖：M-02（历史资料）、M-05（数据分析验证）

---

#### M-04｜IP运营助手

**命名**：`IP运营助手`（Content Strategist）

**核心职责**
- 自媒体选题策划（抖音/小红书/公众号）
- 内容生产（图文、短视频脚本、标题）
- 粉丝互动策略制定
- 数据复盘与内容迭代建议

**产出标准**
- 选题清单：每周≥3条，含话题角度、预计效果
- 内容稿：可直接发布的文案/脚本（含话题标签）
- 运营周报：阅读/互动/粉丝数据摘要
- 交付物命名规范：`[平台]_[类型]_[主题]_v[A-B]_[日期]`

**协作接口**
- 接收来自：主Agent（运营任务）
- 输出给：主Agent
- 协作依赖：M-02（素材库）、M-05（数据反馈）

---

#### M-05｜业务分析师

**命名**：`业务分析师`（Business Analyst）

**核心职责**
- 目标市场与行业研究
- 竞品分析与对标
- 业务数据整理与洞察
- 报告撰写（PPT/Word格式）

**产出标准**
- 市场报告：结构完整，含市场规模、趋势、机会点
- 竞品分析：功能/定价/策略对比表
- 业务建议：含优先级和落地步骤
- 交付物命名规范：`[行业/客户]_分析报告_[日期]`

**协作接口**
- 接收来自：主Agent（分析任务）
- 输出给：主Agent、各专员（业务背景输入）
- 协作依赖：M-02（背景资料）、M-03（政策维度）

## 三、任务派工规范

### 3.1 何时派工

**主Agent判断标准**（满足任一即派工）：

| 条件 | 描述 | 阈值 |
|-----|------|------|
| 复杂度 | 任务可拆解为独立子任务 | 子任务≥2 |
| 专业性 | 任务落在专员能力矩阵内 | - |
| 时长预估 | 预计完成时间>5分钟 | >5min |
| 重复性 | 同类任务已出现过≥2次 | ≥2次 |
| 并行可行性 | 多个子任务可同时执行 | 独立无依赖 |

**直接主Agent执行**（不派工）：
- 纯问答、解释、翻译类任务
- 单一子任务、耗时<2分钟
- 需要主Agent全局判断的综合性任务

### 3.2 派工上下文模板

派工时，主Agent必须传递以下结构化上下文，不得遗漏：

```
【任务派工单】
任务编号：T-[YYYYMMDD]-[序号]
派工时间：[时间]
目标专员：M-0[X] [专员名称]
原始任务：[用户原始需求描述]

=== 任务背景 ===
[为什么做此任务，业务目的或用户动机]

=== 具体要求 ===
1. [明确可验收的子任务描述]
2. [每条要求需含验收标准]

=== 约束条件 ===
- 截止时间：[具体时间]
- 格式要求：[输出格式]
- 参考资料：[已获取的素材或文件路径]

=== 交付标准 ===
[主Agent对subagent结果的验收检查项]

=== 上下文备注 ===
[其他需要专员知晓但不影响任务本身的补充信息]
```

### 3.3 结果汇总流程

```
subagent执行完成
    ↓
主Agent接收结果（subagent结果 + 质量标注）
    ↓
主Agent校验：
  ✅ 内容完整性与派工单一致性
  ✅ 格式规范符合度
  ✅ 是否有明显错误或缺失
    ↓
通过 → 整合入主Agent输出 → 交付用户
不通过 → 标注问题 → 重新派工（注明修改点）
    ↓
主Agent生成最终交付物（整合多专员结果）
```

### 3.4 质量验收标准

| 维度 | 标准 | 检查方法 |
|-----|------|---------|
| 完整性 | 100%覆盖派工单要求 | 对照派工单逐项打勾 |
| 格式 | 符合交付物命名规范 | 文件名检查 |
| 准确性 | 无事实性错误 | 关键数据/法规条目核实 |
| 可用性 | 输出可直接使用或经简单修改后使用 | 主Agent主观评估 |
| 时效性 | 引用数据/法规为最新版本 | 注明引用日期 |

**质量评分**（主Agent内部使用）：
- A（90-100）：直接可用，超出预期
- B（70-89）：基本可用，需小幅修改
- C（50-69）：部分不完整，需返工
- D（<50）：完全不可用，重新执行

## 四、会话Context隔离方案

### 4.1 问题定义

subagent执行长任务时，会向主Agent回传结果，导致主Agent context膨胀（token消耗增加、响应变慢）；同时不同类型任务context混杂，影响后续任务推理质量。

### 4.2 隔离策略

#### 策略A：长任务Session隔离（推荐）

**适用场景**：任务预计>10分钟，或预计产生大量中间产物。

**做法**：
- 主Agent spawn subagent后，subagent在独立session中执行
- subagent通过 `sessions_yield` 返回结果，主Agent session不积累中间过程
- 临时文件写入 `workspace/tmp/[task-id]/`，不进入主Agent context

**Session命名规范**：
```
agent:main:sub:[task-id]
例：agent:main:sub:T-20260324-001
```

#### 策略B：按任务类型分配独立执行路径

| 任务类型 | 推荐隔离程度 | 实现方式 |
|---------|-------------|---------|
| 快速查询（<2min） | 低 | 直接执行，不spawn |
| 标准派工（2-10min） | 中 | spawn，yield返回 |
| 复杂分析（>10min） | 高 | 独立session + 临时文件 |
| 批量处理（>30min） | 极高 | cron化，输出到文件 |

#### 策略C：Context压缩触发

**触发条件**（满足任一）：
- 主Agent session历史消息>100条
- Token预估>上下文窗口50%
- 连续3个任务属于同一专员类型

**操作**：主Agent在派工后，对已完成任务的历史消息做内部摘要归档（不显示给用户），释放context空间。

### 4.3 临时文件清理规范

| 文件类型 | 存储位置 | 保留期限 | 清理规则 |
|---------|---------|---------|---------|
| 中间结果 | `workspace/tmp/[task-id]/` | 任务完成后7天 | 主Agent在结果整合完成后执行 |
| 草稿版本 | `workspace/tmp/[task-id]/drafts/` | 任务完成后3天 | 同上 |
| 最终交付 | `workspace/docs/`（正式目录） | 永久 | 不清理 |
| 错误日志 | `workspace/logs/` | 30天 | 自动清理cron |

**清理命令规范**：
```bash
# 清理指定任务临时文件
rm -rf workspace/tmp/T-YYYYMMDD-XXX/

# 批量清理7天前临时文件（建议cron执行）
find workspace/tmp/ -maxdepth 1 -type d -mtime +7 -exec rm -rf {} +
```

## 五、协作流程设计

### 5.1 完整协作流程

```
┌─────────────────────────────────────────────────────────────┐
│                        用户需求输入                          │
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│                    主Agent：任务分析                         │
│  1. 理解用户需求                                              │
│  2. 判断是否派工（见3.1节）                                    │
│  3. 如需派工：任务分解 + 选择专员                              │
│  4. 生成派工单（见3.2节模板）                                  │
└────────────┬──────────────────────┬────────────────────────┘
             ▼                      ▼
    ┌─────────────┐         ┌─────────────┐
    │  派工执行    │         │  直接执行   │
    │  M-0X专员   │         │  主Agent   │
    └──────┬──────┘         └──────┬──────┘
           ▼                       │
    ┌──────────────────────────────┘
    ▼
┌─────────────────────────────────────────────────────────────┐
│                    主Agent：结果整合                         │
│  1. 收集各专员/自执行结果                                    │
│  2. 质量验收（见3.4节）                                       │
│  3. 内容整合、逻辑串联                                        │
│  4. 生成最终交付物                                           │
└────────────┬────────────────────────────────────────────────┘
             ▼
┌─────────────────────────────────────────────────────────────┐
│                      用户交付                                │
└─────────────────────────────────────────────────────────────┘
```

### 5.2 任务链标准格式

所有多专员任务，主Agent生成任务链文档并跟踪：

```
【任务链：T-[YYYYMMDD]-[序号]】
主题：[任务名称]
创建时间：[时间]
状态：🟡进行中 / ✅完成 / ❌终止

=== 任务分解 ===
1. [子任务A] → M-0X [专员] → [预计时长] → [状态]
2. [子任务B] → M-0X [专员] → [预计时长] → [状态]
...

=== 执行顺序 ===
[并行/串行说明]

=== 当前进度 ===
[最新状态描述]

=== 交付物清单 ===
- [ ] [交付物名称] - [专员] - [状态]
```

### 5.3 交接文档规范

**派工单**（主Agent → 专员）：见3.2节模板

**交付单**（专员 → 主Agent）：subagent返回时附带

```
【交付单：D-[YYYYMMDD]-[序号]】
任务编号：T-[YYYYMMDD]-[序号]
执行专员：M-0X [专员名称]
完成时间：[时间]
质量自评：[A/B/C/D] + 简短说明

=== 产出清单 ===
1. [产出物名称]：[存放路径或内联内容摘要]
2. ...

=== 关键说明 ===
[专员对产出的补充解释、非标准处理说明]

=== 异常记录（如有） ===
[遇到的问题、依赖缺失、假设条件]
```

## 六、身份标注机制

### 6.1 subagent结果来源标注

**标注格式**（主Agent整合时强制添加）：

```
---
📌 信息来源：[专员名称]（M-0X）
完成时间：[时间]
质量评级：[A/B/C/D]
---
[内容]
---
```

**主Agent输出示例**：
> 以下是为您整理的东南亚劳动法对比分析：
>
> ---
> 📌 信息来源：**政策研究员**（M-03）
> 完成时间：13:45
> 评级：A
> ---
> [政策研究员产出内容]
> ---
>
> 结合市场分析，我再补充以下业务建议：
> [主Agent整合内容]

### 6.2 飞书消息身份显示

**飞书消息格式规范**：
